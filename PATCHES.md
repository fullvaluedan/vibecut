Pure functions only; a compact aggregated note (hard-capped to `MAX_LEDGER_NOTE_CHARS`) feeds into the SAME injection seam `taste.ts` already had.

Ledger writes land at the exact points a decision becomes a fact, all in FrameCut-authored files (never in `build-director-proposals.ts`/`run-director.ts`, reserved for the concurrent multi-pass round): `director-plan-store.ts`'s `openCutPanel` (ours) snapshots what THIS plan proposes before any user decision exists; `director-cut-panel.tsx`'s `apply()` folds the final decisions onto that snapshot and persists the appended record onto the project; the same panel's `handleToggle`/`handleSetAll` detect a post-apply un-check and persist a revision update to the latest record. `ai-cut-actions.ts` was not touched.

Storage rides the project the same way `textStyles` did in the text round (see that section above): `TProject` gains an optional `runLedger?: RunLedgerRecord[]` field (type imported from the new ours-authored module), so every project saved before this feature still loads; `storage/service.ts`'s `saveProject` writes it into the serialized literal and `loadProject` reads it back through `normalizeRunLedger` (drops junk records, strips unknown category keys, re-caps a hand-edited file over the limit).

`taste.ts` itself needed a real redesign mid-round: an early version had `buildDirectorTasteNote` read the ledger by calling `EditorCore.getInstance()` directly (a pattern several command modules already use), reasoning that `taste.ts` has no editor/project access of its own. That broke the test suite when run alongside OTHER files that globally `mock.module("@/core", ...)` with a partial fake editor for their own unit tests (`ripple-shift-elements.test.ts`, `split-elements-linkid.test.ts`) - `bun test` shares one module registry across all files in a run, so whichever mock loaded last won, and `taste.ts` crashed reading `.project` off the stub. Fixed by flipping to a PUSH model: `taste.ts` gained a plain (unpersisted) `ledgerNote` string field and a `setLedgerNote` setter, and `director-dock-shell.tsx` (ours, always mounted for the life of the editor page) subscribes to the active project's `runLedger` via `useEditor` and pushes the derived note in on every change (project load/switch, apply, or a post-apply revision). `taste.ts` still imports the category list/labels/op-category-resolution from `run-ledger.ts` (now the source of truth for both), but has zero editor/`@/core` coupling. `deriveTasteNote`'s signature and behavior are unchanged (still tested standalone with a plain `DirectorTasteStats` argument); `buildDirectorTasteNote()` stays a zero-arg call, so every existing caller needed no change.
