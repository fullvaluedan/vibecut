## Line 1 — AI Director cut quality (the main work)

### What exists now: the golden-footage eval

`bun scripts/director-eval.ts --llm` runs the app's **actual** proposal pipeline
(the extracted pure `buildDirectorProposals` module — same code the editor runs)
with all three live LLM passes, and scores the proposals against ground truth
derived from Dan's finished edits. No footage leaves the machine; only compressed
audio goes to Groq for transcription. LLM responses cache under
`apps/web/.eval-cache/` so re-scoring after non-LLM changes is free.

- **Fixtures** (`apps/web/eval-fixtures/`, gitignored): google-omni (removes 58.5%),
  hermes-cloud (39.4%), how-to-edit (79.6%), pokemon-tcg (32.5%). Dan's editing
  ratio varies wildly per video — a single global setting will never fit.
- **Reference finals come from `Videos/_Finals/` only** (provenance rule — an
  intermediate reference caused a wrong verdict once; see findings addendum 1).
- **Build a new fixture:** `GROQ_API_KEY=... FFMPEG=<path> bun scripts/director-eval-prepare.ts --raw-dir "<OBS clips folder>" --final "Videos/_Finals/<final>.mp4" --name <slug>`

### The scorecard (two views)

- **AUTO** = ops a one-click apply would execute (defaultAccept true).
- **OFFERED** = every review row.
- Headline metrics: **cut recall** (of Dan's cuts, how many we propose),
  **essential words lost** (kept dialog our cuts would destroy — bar is 0),
  **missed cut words**, mean boundary error.

### Current numbers (aggregate across 4 fixtures, latest)

|---|---|---|

### The verdict (findings Addendum 2, `docs/2026-07-11-director-eval-findings.md`)

**No in-app default was changed** — the measurement decides, and it said keep
today's conservative defaults:
- Quality keeper only marginally beats keep-last (essential-lost -6.6%, recall
  -1.5pp); KTD3 requires the scorecard to *clearly* overturn a live-tested default.
- Compression buys +10pp recall on google-omni but +143 essential-lost; unsafe to
  auto-apply at the measured ratio. Stays opt-in behind `--compression`.

**Success bars (R6) MISSED, stated plainly:** essential-lost 106-558/fixture (bar
20); google-omni recall peaks 36.6% (bar 40%).

**Root cause named:** essential-lost is **span-extent dominated** (mean boundary
error 3-9s — the LLM chooses cut *spans* materially different from Dan's), so U1's
sub-frame edge refinement (correct, active in every combo) cannot move the
aggregate. The lever is span granularity, not edges.

### THE NEXT PLAN (recommended, not yet started)

The one move that closes the gap fastest, both deferred in the last plan:
1. **Word-granularity cut spans + span clamping**: make the plan pass emit
   word-level cut extents, or clamp LLM cut spans to Dan-sized removals. Directly
   attacks essential-words-lost (the bar-zero metric).
2. **Dedicated retake-hunt recall pass** at word granularity, surfacing retakes /
   false-starts as **OFFERED** rows (not AUTO). The safe way to close the
   under-cutting gap without paying the compression essential-lost cost.
3. Note: how-to-edit (79.6% removal) got *worse* recall with compression — extreme
   footage needs the take/recall structure, not a bigger prompt number.

Aggressive knobs (`--keeper quality`, `--compression`) are now measured and
available; not default.
